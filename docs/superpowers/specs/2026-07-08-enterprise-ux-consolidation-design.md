# 企业级体验收敛设计规格

## 背景

当前桌面版已经可以安装和运行，也具备凭证、报表、税务、银行、资产、薪酬、更新中心等大量能力。但用户进入软件后会感到功能未完善、界面繁杂。截图审计显示，主要问题不是单个组件样式，而是信息架构过长、模块边界不清、功能成熟度未显式区分、导航落点不稳定。

本规格采用“方案 A：先改体验骨架，再补关键闭环”。目标是让软件从功能展示页变成财务经理每天可使用的工作台。

## 目标

1. 降低首屏认知负担，让用户打开软件后立刻知道今天要处理什么。
2. 简化左侧导航，把页面从长滚动功能合集收敛为少量清晰工作区。
3. 明确区分正式可用、演示数据、待配置和待接入能力，避免误导用户。
4. 修复菜单锚点和滚动落点，让导航行为稳定可靠。
5. 补齐首次使用和更新配置的基础闭环，减少用户不知道从哪里开始的问题。

## 非目标

1. 本轮不接入真实税局、银行、OCR 或发票验真服务。
2. 本轮不重写所有业务模块的底层核算逻辑。
3. 本轮不把所有内存 MVP 一次性迁移到持久化数据库。
4. 本轮不引入新的 UI 框架，继续使用 React、TypeScript、Ant Design 和现有样式体系。

## 目标用户与使用场景

目标用户是中国企业财务经理、财务主管和老板视角的经营管理者。

核心场景：

- 早上打开软件查看今天需要处理的财务事项。
- 月末查看哪些凭证、对账、归档、报表和税务事项还没有完成。
- 进入某个模块时，只看到当前模块最重要的状态和动作。
- 当能力尚未配置或只是演示数据时，软件明确提示边界。

## 产品结构

### 一级工作区

左侧导航收敛为 6 个一级入口：

1. 工作台
   - 今日任务、风险提醒、关键指标、常用动作。
2. 财务核算
   - 凭证、账簿、往来、银行对账、固定资产、薪酬、存货、预提摊销、期间结账。
3. 报表与分析
   - 财务报表、报表映射、归档快照、合并报表、经营分析报告。
4. 税务与合规
   - 税务台账、申报底稿、法规库、风险闭环。
5. 智能助手
   - AI 财务问答、凭证草稿、自动审核、解释风险。
6. 系统设置
   - 公司信息、账套与期间、权限审计、更新中心、外部服务配置。

### 二级入口

每个一级工作区内部使用标签或分组列表呈现二级模块。默认只展示最常用的 3 到 5 个入口，更多模块收进“更多功能”区域。

## 首页设计

首页从“指标展示”改为“今日财务任务台”。

首屏顺序：

1. 今日状态条
   - 显示账套、期间、数据状态、更新状态。
2. 今日待办
   - 待审核凭证、待对账流水、待归档文件、待月结检查、待复核风险。
3. 高优先级风险
   - 只展示 3 条最重要风险，其余进入风险中心。
4. 关键指标
   - 收入、利润、现金流、税负率、风险等级保留，但视觉权重低于待办。
5. 常用动作
   - 导入数据、生成报表、执行月结、打开 AI 助手、检查更新。

首页不再展示大段产品说明。说明文字只保留一句状态摘要，详细解释放入帮助或模块内部。

## 模块页设计

每个模块页统一为三层：

1. 模块头
   - 标题、成熟度标签、当前账套/期间、主要动作。
2. 工作区摘要
   - 最多 3 个状态项，展示“需要处理的问题”，不再铺满 4 到 6 张卡片。
3. 详细内容
   - 表格、表单、图表、历史记录放在标签页内。

模块说明文字不超过两行。冗长边界说明放到“能力边界”折叠区。

## 功能成熟度标签

新增统一的成熟度模型：

- `ready`：正式可用，使用真实本地数据或正式分录。
- `demo`：演示数据，适合体验流程，不代表企业真实账。
- `needs_config`：需要配置，例如更新源、OCR 服务、政策库地址。
- `planned`：已规划但尚未接入。

所有模块、主要按钮和空状态都应能显示成熟度。待接入功能不能用普通主按钮误导用户；应显示为次级按钮或禁用按钮，并给出下一步配置提示。

## 首次启动向导

首次启动向导只做最小闭环，避免变成复杂实施系统。

步骤：

1. 公司信息
   - 公司名称、行业、默认账套名称。
2. 会计期间
   - 当前期间、是否启用月结控制。
3. 数据模式
   - 使用演示数据或创建空账套。
4. 更新配置
   - 政策包 manifest URL、软件本体 manifest URL，可跳过。

向导完成后写入本地配置。用户可在系统设置中重新打开。

## 更新中心改进

更新中心从“状态展示”升级为“配置 + 检查 + 安装”闭环：

- 支持在界面中保存政策包和软件本体 manifest URL。
- 显示上次检查时间、下次计划检查时间和失败原因。
- 软件本体更新包下载完成后显示“安装更新”按钮。
- 安装动作调用 `ChinaFinanceUpdater.exe`，主程序不自我覆盖。
- 未配置更新源时，显示“未配置”，不显示为错误。

## 导航与滚动行为

禁止依赖长页面锚点作为主要导航体验。第一轮可以保留现有单页结构，但必须修复：

- 点击左侧一级入口后滚动到正确工作区顶部。
- 当前菜单项根据可见工作区更新选中状态。
- 同一页面内不出现多个容易混淆的滚动容器。
- 数据中心、企业设置等入口不得落到期间结账或报表内容中间。

如果实现成本可控，优先引入轻量工作区状态：左侧菜单切换 `activeWorkspace`，主内容只渲染当前工作区。

## 视觉原则

- 面向 SaaS/财务软件，采用安静、克制、可扫描的界面。
- 减少大面积装饰、重复卡片和冗长文案。
- 卡片只用于独立信息块，不做卡片套卡片。
- 操作按钮按主次排序：每个模块最多一个主按钮。
- 表格和任务列表优先于装饰性指标卡。
- 保留 Ant Design 组件，但统一间距、字号、标签和按钮密度。

## 数据与配置

前端新增配置视图，后端需要提供本地配置读写接口。配置保存到用户数据目录，不写入源码目录。

配置项：

- 公司名称。
- 默认账套。
- 当前期间。
- 是否完成首次启动。
- 政策包更新源。
- 软件本体更新源。

敏感信息不在本轮保存。若未来接入银行、税局或第三方 OCR 凭据，必须另建凭据存储方案。

## 错误处理与空状态

- 未配置外部服务：显示“待配置”，附带进入设置的按钮。
- 当前功能只支持演示数据：显示“演示数据”，说明不会影响正式账。
- 接口失败：显示可重试状态，不把错误堆栈暴露给用户。
- 导航落点失败：回退到工作区顶部。

## 测试要求

后端：

- 配置读写接口测试。
- 首次启动状态测试。
- 更新源保存和读取测试。

前端：

- 导航分组和默认工作区测试。
- 成熟度标签渲染测试。
- 首次启动向导测试。
- 更新中心配置和安装按钮测试。
- 关键页面文案收敛测试，避免重新出现长篇说明。

视觉验证：

- 首页、财务核算、系统设置三个视口截图。
- 检查菜单选中状态、按钮不溢出、文本不重叠。
- 检查数据中心和企业设置入口不再落到错误模块。

## 分阶段实施

第一阶段：体验骨架

- 重构导航和首页。
- 增加成熟度标签。
- 收敛模块头部和说明文字。
- 修复工作区切换与滚动落点。

第二阶段：首次启动和设置

- 新增本地配置模型和 API。
- 新增首次启动向导。
- 新增系统设置中的公司、账套、期间和更新源配置。

第三阶段：更新中心闭环

- 在 UI 中保存更新源。
- 添加“安装更新”动作。
- 明确下载成功、校验失败、未配置和等待安装状态。

## 验收标准

1. 新用户首次打开软件时能通过向导进入工作台。
2. 首页首屏能看到今日待办、风险和常用动作，不需要先理解全部模块。
3. 左侧导航点击后内容落点稳定，菜单选中状态正确。
4. 未完善功能都有明确成熟度标签，不再像正式功能一样呈现。
5. 系统设置可以配置政策包和软件更新源。
6. 更新中心在未配置时不报错，配置后能检查，下载后能提供安装入口。
7. 后端测试、前端测试和生产构建通过。

## 自查结果

- 无占位项。
- 设计范围聚焦在体验骨架、首次启动和更新中心闭环。
- 不承诺本轮接入真实银行、税局、OCR 或发票验真。
- 与现有 React、Ant Design、FastAPI 和本地桌面发行包架构一致。
