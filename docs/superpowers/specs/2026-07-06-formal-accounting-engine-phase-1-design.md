# 正式会计核算引擎一期设计规格

## 1. 背景

当前项目已经具备凭证中心、账簿读模型、会计期间状态、固定资产、工资管理和财务报表自动生成 MVP。现有能力主要验证产品流程：凭证可新增、审核、过账、反过账，账簿和报表可基于已审核凭证生成演示结果。

下一阶段不再继续扩大演示功能，而是建设正式会计核算底座。正式核算版本的核心变化是：业务凭证不再直接承担账务事实，正式账务事实由不可变的会计分录承载；账簿、报表、期间控制、多账套和多币种都围绕正式分录展开。

一期目标是建立最小但正确的核算内核，让现有凭证中心可以正式过账为分录，并让账簿读模型从正式分录取数。后续多币种重估、期末损益结转、辅助核算、电子凭证归档和正式报表映射都在该内核上继续扩展。

## 2. 设计目标

- 建立独立的正式核算内核，不直接推翻现有 MVP 凭证中心。
- 新增不可变 `journal_entry` 和 `journal_line` 作为正式账务事实。
- 将凭证中心的“过账”升级为生成正式会计分录。
- 已过账分录不允许原地修改或删除，后续更正通过冲销、调整或反向分录完成。
- 总账、明细账、科目余额表优先从正式分录读取。
- 财务报表优先从正式分录读取；没有正式分录时才回退现有演示数据。
- 数据模型从一期开始包含账套、会计期间、科目、币种、汇率和审计字段。
- 保留现有 MVP 演示链路，但在文档和接口响应里清晰标记来源。

## 3. 非目标

- 一期不实现完整外币期末重估和汇兑损益自动结转。
- 一期不实现完整辅助核算维度管理。
- 一期不实现固定资产折旧、工资计提、税费计提自动生成正式分录。
- 一期不实现损益结转、反结账审批流或会计档案归档流程。
- 一期不实现完整现金流量表自动拆分。
- 一期不替代企业正式财务负责人、会计主管或审计人员的专业判断。

## 4. 准则与合规约束

正式核算内核按中国企业会计准则方向设计，主要约束如下：

- 会计信息必须可追溯、可复核、不可随意篡改。
- 过账后的正式分录应保留来源凭证、操作人、过账时间、账套、期间和审计链。
- 会计期间关闭后不得继续新增普通过账分录；确需调整时应通过重开期间或后续期间调整流程处理。
- 多币种数据应同时保留交易币种金额、汇率、本位币金额，避免只保存折算后金额导致追溯困难。
- 电子凭证、附件和归档能力后续需要对齐电子凭证会计数据标准和会计档案管理要求。

## 5. 推荐架构

一期采用“并行正式内核”架构。

现有凭证中心继续作为录入和审核层，负责业务字段、附件记录、AI 自动审核和人工审核状态。正式核算内核新增独立服务和持久化表，负责将已审核凭证转换为正式分录，并为账簿、报表和期间控制提供统一数据源。

```text
凭证中心
  -> 审核
  -> 正式过账服务
  -> journal_entry / journal_line
  -> 总账 / 明细账 / 科目余额表 / 报表
```

设计理由：

- 保留当前产品可用性，避免一次性重写凭证中心。
- 正式核算事实与 MVP 工作流状态分离，边界更清楚。
- 后续固定资产、工资、税务、电商平台导入也可以直接生成正式分录，不必绕回凭证中心。
- 账簿和报表逐步迁移到正式分录，减少演示逻辑对核心财务口径的污染。

## 6. 核心领域模型

### 6.1 账套

账套用于隔离公司、核算主体、会计准则、默认本位币和期间状态。

关键字段：

- `id`
- `name`
- `base_currency`
- `accounting_standard`
- `status`
- `created_at`
- `updated_at`

一期沿用现有 `default` 和 `cross_border` 两个账套，同时为正式核算表增加账套字段。

### 6.2 会计期间

会计期间用于控制过账和结账。

关键字段：

- `account_set_id`
- `period`
- `status`: `open`、`closed`
- `closed_by`
- `closed_at`

一期继续复用现有期间接口，但关闭逻辑要从“检查未过账凭证”升级为“检查正式过账完整性”。若期间已关闭，正式过账服务拒绝新增普通分录。

### 6.3 科目体系

科目体系用于规范分录行的科目编码、名称、类别和余额方向。

关键字段：

- `account_set_id`
- `account_code`
- `account_name`
- `account_type`: `asset`、`liability`、`equity`、`revenue`、`cost`、`expense`
- `normal_balance`: `debit`、`credit`
- `is_active`

一期先内置企业会计准则常用科目，覆盖现有凭证和报表需要的编码：`1001`、`1002`、`1122`、`1405`、`1601`、`2001`、`2202`、`2221`、`4001`、`6001`、`6051`、`6401`、`6601`、`6602`、`6603`。

### 6.4 币种与汇率

币种和汇率用于为多币种正式核算打地基。

关键字段：

- `currency_code`
- `currency_name`
- `decimal_places`
- `account_set_id`
- `rate_date`
- `source_currency`
- `target_currency`
- `rate`

一期默认支持 `CNY`、`USD`、`EUR`、`HKD`。现有凭证没有外币字段时，默认按账套本位币入账，汇率为 `1`。外币凭证后续扩展时必须保存原币金额和本位币金额。

### 6.5 正式分录头

`journal_entry` 是正式账务事实的头表。

关键字段：

- `id`
- `account_set_id`
- `period`
- `entry_date`
- `entry_number`
- `source_type`: `voucher_center`、`fixed_asset`、`payroll`、`tax`、`manual_adjustment`
- `source_id`
- `description`
- `status`: `posted`、`reversed`
- `base_currency`
- `created_by`
- `posted_by`
- `posted_at`
- `reversal_of_entry_id`

一期只允许创建 `posted` 状态分录。撤销正式分录时不删除原记录，而是创建冲销分录并把原分录标记为已冲销。

### 6.6 正式分录行

`journal_line` 是正式借贷分录行。

关键字段：

- `id`
- `journal_entry_id`
- `line_no`
- `account_code`
- `account_name`
- `direction`: `debit`、`credit`
- `currency`
- `original_amount`
- `exchange_rate`
- `base_amount`
- `description`

一期要求每张正式分录借贷本位币金额相等，否则拒绝过账。原币金额、汇率、本位币金额都使用 Decimal 保存并统一保留两位小数，汇率保留足够精度。

## 7. 核心流程

### 7.1 凭证正式过账

流程：

1. 用户在凭证中心新增凭证。
2. AI 自动审核给出风险提示。
3. 人工审核凭证。
4. 用户触发过账。
5. 正式过账服务校验账套、期间、科目、借贷平衡和重复过账。
6. 服务生成 `journal_entry` 和 `journal_line`。
7. 凭证中心记录正式分录 ID 和过账状态。
8. 审计日志记录来源凭证、操作人、分录号和期间。

关键规则：

- 未审核凭证不能正式过账。
- 已过账凭证不能重复生成正式分录。
- 期间关闭后不能过账普通凭证。
- 分录行科目必须存在且启用。
- 借贷本位币合计必须相等。

### 7.2 正式反过账

一期不允许直接删除正式分录。

反过账流程：

1. 用户对已过账凭证触发反过账。
2. 系统校验期间未关闭。
3. 系统生成一张冲销分录，借贷方向与原分录相反。
4. 原分录标记为 `reversed`，冲销分录记录 `reversal_of_entry_id`。
5. 凭证中心状态更新为未过账。

该设计保证账务轨迹完整，不会因为反过账而丢失历史。

### 7.3 账簿读取

总账、明细账和科目余额表优先读取正式分录：

- 总账按账套、期间、科目聚合 `base_amount`。
- 明细账按科目列出正式分录行和来源凭证。
- 科目余额表按科目类型和余额方向计算期末余额。

如果当前环境没有任何正式分录，可保留现有演示回退，但响应要标记 `source="mvp_voucher_workflow"` 或 `source="sample_finance_data"`。

### 7.4 财务报表取数

财务报表服务优先读取正式分录生成报表。

一期继续沿用现有报表项目逻辑，但数据源改为正式科目余额和发生额：

- 资产负债表读取资产、负债、权益类科目余额。
- 利润表读取收入、成本、费用类科目发生额。
- 现金流量表一期保留基础占位项目，后续通过现金流项目映射增强。
- 所有者权益变动表一期读取净利润和权益余额，后续补完整权益变动项目。

## 8. API 设计

一期新增或调整以下接口：

```text
GET /api/v1/accounting/accounts?account_set_id=default
GET /api/v1/accounting/journal-entries?account_set_id=default&period=2026-06
GET /api/v1/accounting/journal-entries/{entry_id}
POST /api/v1/vouchers/center/{voucher_id}/post
POST /api/v1/vouchers/center/{voucher_id}/unpost
GET /api/v1/ledger/general?period=2026-06&account_set_id=default
GET /api/v1/ledger/detail?period=2026-06&account_code=6602&account_set_id=default
GET /api/v1/ledger/account-balances?period=2026-06&account_set_id=default
POST /api/v1/financial-statements/generate
```

`/post` 和 `/unpost` 仍保留现有路径，但内部语义升级为正式分录生成和冲销。新增 `accounting` 路由用于读取正式科目和正式分录。

## 9. 权限与审计

新增权限：

- `accounting.entry.read`
- `accounting.entry.post`
- `accounting.entry.reverse`
- `accounting.account.read`

沿用或增强权限：

- `voucher.post`
- `voucher.unpost`
- `ledger.read`
- `statement.generate`
- `ledger.period.manage`

审计事件：

- `accounting.entry.post`
- `accounting.entry.reverse`
- `accounting.entry.read`
- `accounting.account.read`

所有正式过账、冲销、期间关闭和报表生成都必须记录操作人、账套、期间、来源 ID 和结果状态。

## 10. 数据迁移与兼容

一期不迁移历史 MVP 凭证为正式分录。用户可以对现有已审核凭证重新触发过账，生成正式分录。

兼容策略：

- 现有凭证中心字段继续保留。
- `posting_status` 继续展示前端状态。
- 新增正式分录 ID 字段用于关联过账结果。
- 账簿读模型先读取正式分录；若没有正式分录，再按现有逻辑读取已审核凭证作为兼容回退。
- 报表生成同样优先正式分录，再回退已审核凭证或样例数据。

## 11. 前端体验

一期前端不做大改版，只在现有面板中显式区分正式核算来源：

- 凭证中心显示“正式分录号”或“未正式过账”。
- 账簿面板显示数据来源：正式分录、MVP 凭证工作流或样例数据。
- 财务报表面板显示数据来源和资产负债平衡状态。
- 系统管理面板能看到正式过账和冲销审计日志。

## 12. 测试策略

后端测试覆盖：

- 未审核凭证不能正式过账。
- 已审核且借贷平衡凭证可生成正式分录。
- 重复过账被拒绝。
- 已关闭期间不能正式过账。
- 反过账生成冲销分录，不删除原分录。
- 总账、明细账、科目余额表从正式分录取数。
- 财务报表优先从正式分录取数。
- 正式过账、冲销、读取和权限拒绝都写入审计日志。

前端测试覆盖：

- 凭证中心展示正式过账结果。
- 账簿面板展示正式分录来源。
- 财务报表面板展示正式分录来源。
- 权限或期间错误时显示后端错误信息。

## 13. 后续路线

二期：完整多币种核算，包括外币凭证录入、汇率表、原币账簿和本位币折算。

三期：辅助核算维度，包括客户、供应商、员工、部门、项目、资产、平台和 SKU。

四期：期末处理，包括折旧计提、工资计提、税费计提、损益结转和汇兑损益。

五期：正式报表映射配置，包括报表项目、科目映射、现金流项目和管理报表模板。

六期：电子凭证和会计档案，包括来源文件、验真状态、归档状态、保管期限和电子凭证数据标准适配。

## 14. 验收标准

- 凭证中心已审核凭证可以正式过账为不可变分录。
- 正式分录借贷平衡校验失败时拒绝过账。
- 正式分录生成后，总账、明细账和科目余额表能从正式分录读取。
- 反过账生成冲销分录，而不是删除原分录。
- 期间关闭后普通过账被拒绝。
- 财务报表优先使用正式分录来源。
- 前端能看到正式核算来源和正式分录号。
- 所有关键操作都有审计日志。
